The Boss × Universal Agent Runtime

A complete design plan for wiring the UAR at uar.travisjames.ai as the native provider backend for The Boss — with Kratos-managed identity, a free-tier model for members, and first-class BYOK support for all other providers.

§ 01 — Architecture

System Architecture

Three systems interact to produce the complete experience. The Boss (desktop client) handles all UI and provider routing. Kratos manages identity and issues session JWTs. The UAR serves as both the authenticated AI backend for members and the routing layer that enforces tier policy.

§ 02 — Authentication

Auth Flow

Authentication is Kratos-native. The Boss embeds a lightweight auth state manager that opens the Kratos self-service login UI in a webview when the user attempts to use the UAR provider without a valid session. On success, Kratos returns a session token which The Boss exchanges for a short-lived JWT to use as the UAR Bearer credential.

User clicks UAR provider
→
Check stored session token
→
No token / expired
→
Open Kratos login UI
Login success
→
Store session cookie + JWT
→
Inject JWT as apiKey
→
UAR request authenticated

Token Storage Strategy

Electron's safeStorage API encrypts the Kratos session token on disk. The JWT is derived fresh from the session on each provider initialization, never stored independently. On app launch, the session is silently validated against https://kratos.travisjames.ai/sessions/whoami — if valid, the UAR provider activates automatically with no user interaction required.

Session framing: The login UI should present as "Sign in to your travisjames.ai membership" — not as an API credential screen. This positions The Boss as the member client for the travisjames.ai community and sets up the upgrade path toward paid tiers naturally.

§ 05 — Phase 2

Kratos Auth Integration

A new KratosAuthService handles the full session lifecycle: silent re-validation on startup, initiating login flows, and injecting the JWT into the travisai provider's apiKey field at runtime.

2a — KratosAuthService

New file: src/renderer/src/services/KratosAuthService.ts

TypeScript KratosAuthService.ts — New file 
const KRATOS_BASE = 'https://kratos.travisjames.ai'
const SESSION_KEY = 'travisai_session_token'

export class KratosAuthService {
  private sessionToken: string | null = null
  private jwt: string | null = null

  /** Called at app boot — silently restores session if valid */
  async initialize(): Promise<boolean> {
    const stored = await window.api.safeStorage.get(SESSION_KEY)
    if (!stored) return false
    return this.validateSession(stored)
  }

  /** Validate session against /sessions/whoami */
  async validateSession(token: string): Promise<boolean> {
    const res = await fetch(`${KRATOS_BASE}/sessions/whoami`, {
      headers: { 'X-Session-Token': token }
    })
    if (!res.ok) { this.clearSession(); return false }
    const session = await res.json()
    this.sessionToken = token
    this.jwt = session.identity?.metadata_public?.uar_jwt ?? token
    await this.injectIntoProvider()
    return true
  }

  /** Inject JWT as apiKey on the travisai provider in Redux store */
  private async injectIntoProvider(): Promise<void> {
    if (!this.jwt) return
    store.dispatch(updateProvider({
      id: 'travisai',
      apiKey: this.jwt
    }))
  }

  /** Trigger login flow in a BrowserWindow overlay */
  async initiateLogin(): Promise<boolean> {
    // Opens Kratos self-service login URL in modal webview
    // Resolves promise when Kratos redirects to the success callback URL
    return window.api.auth.openKratosLogin(KRATOS_BASE)
  }

  clearSession(): void {
    this.sessionToken = null
    this.jwt = null
    window.api.safeStorage.delete(SESSION_KEY)
    store.dispatch(updateProvider({ id: 'travisai', apiKey: '' }))
  }

  isAuthenticated(): boolean { return !!this.jwt }
  getJwt(): string | null { return this.jwt }
}

2b — Main Process IPC Handler

The login flow requires a BrowserWindow modal in the main process. A minimal IPC channel handles the handshake.

TypeScript src/main/auth/kratosLogin.ts — New IPC handler 
// Main process handler
ipcMain.handle('auth:openKratosLogin', async (event) => {
  const win = new BrowserWindow({
    width: 480, height: 640,
    parent: BrowserWindow.fromWebContents(event.sender)!,
    modal: true, show: false,
    webPreferences: { nodeIntegration: false, contextIsolation: true }
  })

  // Kratos self-service login URL
  const loginUrl = `https://kratos.travisjames.ai/self-service/login/browser`
  win.loadURL(loginUrl)
  win.once('ready-to-show', () => win.show())

  // Intercept the post-login redirect to extract the session token
  win.webContents.on('will-navigate', async (e, url) => {
    if (url.startsWith('theboss://auth/callback')) {
      const token = new URL(url).searchParams.get('session_token')
      if (token) {
        await safeStorage.set(SESSION_KEY, token)
        win.close()
        // Resolve with token so renderer can validate and inject
      }
    }
  })
})
Kratos config requirement: The Kratos instance at kratos.travisjames.ai must have theboss://auth/callback registered as an allowed return URL in its selfservice.allowed_return_urls config. The Electron app registers the theboss:// protocol handler via app.setAsDefaultProtocolClient('theboss').
§ 06 — Phase 3

Login UI — Membership Framing

The login experience should feel native to The Boss — not like an external auth step. It's an onboarding moment that frames the free tier as a membership in the travisjames.ai community, not as an API key entry form.

Login Entry Point 2
First Unauthenticated Request
If the user selects a travisai model and sends a message without a session, the chat stream is intercepted, the login modal opens, and on success the message is automatically resent — seamless from the user's perspective.

§ 07 — Phase 4

Free Model Configuration

The UAR at uar.travisjames.ai exposes a model named free (or a more descriptive ID like travisai/free) that routes authenticated free-tier requests to your candle-vllm instance running on the L4 or 4070 Ti. The concrete model behind this ID can be swapped without any client changes.

Client Side
What The Boss sends
Model ID: free
Host: https://uar.travisjames.ai/v1
Auth: Authorization: Bearer {kratos_jwt}
UAR Side
What the UAR does
Validates JWT via Kratos JWKS endpoint.
Checks Cedar policy: tier = "free" → allow model "free".
Routes to candle-vllm via LiterLlm.

Model Discovery via /v1/models

Cherry Studio's "auto-fetch models" button calls GET /v1/models on the provider. The UAR should return a model list appropriate to the authenticated user's tier — free users see only the free model, paid users see all available models. This creates a natural upsell moment inside The Boss's own model selector.

§ 08 — Phase 5

BYOK Surface

BYOK (Bring Your Own Key) for all providers other than travisai requires no changes — this is the stock Cherry Studio behavior. The Boss ships with OpenAI, Anthropic, Gemini, Ollama, and all other providers fully intact. Users who want to use their own Claude or OpenAI key simply configure those providers exactly as they would in unmodified Cherry Studio.

The only meaningful surface change is in the onboarding flow: the default provider shown to new users should be travisai (not OpenAI), and the Settings → Model Services panel should list travisjames.ai first, with a clear visual separator between "Your membership" and "Bring your own key" sections.

Provider settings layout change: Add a visual section header system to the provider list in src/renderer/src/pages/settings/ProviderSettings.tsx (or equivalent). Two groups: "travisjames.ai" (pinned at top, shows membership status) and "Your keys" (all other providers, collapsed by default for new users).

§ 11 — UAR Backend

JWT Validation Middleware

The UAR Axum middleware validates the Bearer JWT from Kratos by fetching the JWKS from Kratos' /.well-known/jwks.json endpoint. The decoded claims carry the user's identity and tier, which Cedar policy uses to determine which models the request may access.

Rust uar/src/middleware/kratos_auth.rs — Pseudocode structure 
// Axum extractor — runs before every /v1/* handler
pub struct KratosIdentity {
    pub subject: String,         // Kratos identity ID
    pub email: String,
    pub tier: MemberTier,        // Free | Member | Pro
}

// Cedar principal built from identity:
// principal = Member::{identity_id}
// context  = { tier: "free", model_requested: "free" }
// action   = Action::"ChatCompletion"
// resource = Resource::"free"

// Cedar policy (free tier):
// permit(principal, action == Action::"ChatCompletion", resource == Resource::"free")
// when { context.tier == "free" };

// Cedar policy (paid tier):
// permit(principal, action == Action::"ChatCompletion", resource)
// when { context.tier in ["member", "pro"] };
§ 12 — UAR Backend

Tier Enforcement Strategy

Tier — Free
travisjames.ai Member (Free)
Model access: free only (candle-vllm, local open-weight model)
Rate limit: configurable per-user quota
Context window: capped (e.g., 8k tokens)
Knowledge base: supported (local embeddings)
Agents: basic tool use
Tier — Pro ($20+/mo)
Paid Member
Model access: all UAR-hosted models
Higher rate limits + larger context windows
Cloud storage + cross-device sync
Priority queue on the parking lot scheduler
Set via Kratos identity trait: tier: "pro"
Tier propagation: When a user upgrades via the travisjames.ai website, Kratos updates their identity traits. The next time The Boss calls /sessions/whoami, the new tier is present in the session. KratosAuthService.validateSession() re-injects the updated JWT, and the UAR's model list response expands automatically. No re-login required.
